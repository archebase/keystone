Keystone / Synapse Development Design

Episode 通用质检与 MCAP 完整性检查

在 episode 详情页提供手动触发的轻量质检入口。当前首个检查项为 MCAP 头尾 magic 校验；未来 Python 脚本型质检通过同一接口和落库模型接入。

背景

部分异常会导致上传后的 MCAP 无法播放，例如播放器初始化时报错：Expected MCAP magic '89 4d 43 41 50 30 0d 0a', found ...。当前错误通常在预览加载早期暴露，不需要等待全量数据解析。

因此需要在 keystone 和 synapse 中提供一个轻量、手动触发、可扩展的质检能力。当前优先检查 MCAP 容器边界是否满足基本格式要求，并将失败结果沉淀到 episode 状态，供后续预览、下载、上传和同步统一拦截。

边界说明：头尾 magic 校验只能证明 MCAP 文件边界基本正确，不能证明文件一定可播放。内部 chunk、schema、压缩数据、CRC 或索引仍可能损坏。该检查用于快速拦截明显坏包。

目标与非目标

目标

  • 在 episode 详情页提供固定按钮 MCAP 质检。
  • 通过通用 API 触发质检项，当前检查项为 mcap_magic。
  • 只读 MCAP 头 8 字节和尾 8 字节，保持轻量。
  • 任何检查失败都将 episodes.qa_status 置为 failed。
  • 失败状态阻止 MCAP 预览、MCAP 下载和云同步。
  • 为后续 Python 脚本质检预留内部扩展结构。

非目标

  • 不新增 episode 状态字段。
  • 不做全量 MCAP 解析。
  • 不在通过头尾校验后自动批准 episode。
  • 不在后续检查通过时自动从 failed 恢复。
  • 不立即实现 Python 脚本执行器。

数据模型复用

本设计不新增字段，复用现有 episodes 和 qa_checks。

字段/表 用途 规则
episodes.qa_status episode 当前对外质量状态 任何检查失败都设置为 failed；检查通过不自动改为 approved
episodes.quality_flag 面向研究员和运营人员的质量说明 检查失败时写入失败摘要，例如头尾 magic 不匹配
qa_checks 记录每次检查的历史和证据 每次点击都插入一条记录，不覆盖旧记录
qa_checks.check_metadata 结构化检查详情 记录 expected/head/tail/file_size 等数据

状态规则

失败

插入 qa_checks.passed=false，设置 qa_status=failed，写入 quality_flag。

通过

插入 qa_checks.passed=true，不自动批准，也不自动恢复 failed。

已同步数据

如果检查失败，不回滚 cloud_synced；后续重新同步必须被阻止。

API 设计

运行质检

POST /api/v1/episodes/:id/qa-checks
Content-Type: application/json

{
  "check_name": "mcap_magic"
}

响应示例：通过

{
  "episode_id": 123,
  "check_name": "mcap_magic",
  "passed": true,
  "score": 1,
  "details": "MCAP head and tail magic matched",
  "check_metadata": {
    "expected_magic": "89 4d 43 41 50 30 0d 0a",
    "found_head_magic": "89 4d 43 41 50 30 0d 0a",
    "found_tail_magic": "89 4d 43 41 50 30 0d 0a",
    "file_size_bytes": 123456789
  },
  "checked_at": "2026-06-05T10:30:00Z"
}

响应示例：失败

{
  "episode_id": 123,
  "check_name": "mcap_magic",
  "passed": false,
  "score": 0,
  "details": "MCAP integrity check failed: tail magic mismatch",
  "check_metadata": {
    "expected_magic": "89 4d 43 41 50 30 0d 0a",
    "found_head_magic": "89 4d 43 41 50 30 0d 0a",
    "found_tail_magic": "8b ef b8 75 c6 97 96 61",
    "file_size_bytes": 123456789
  },
  "checked_at": "2026-06-05T10:30:00Z"
}

错误处理

场景 状态码 说明
episode 不存在 404 返回 {"error":"episode not found"}
未知检查项 400 返回 {"error":"unsupported qa check"}
S3/MinIO 不可用 502 或 503 不写入失败质检记录，避免把基础设施故障误判成数据坏包
文件格式检查失败 200 返回 passed=false，并落库更新 episode 状态

后端实现

执行流程

1 接收请求

解析 episode ID 和 check_name。

2 加载 episode

读取 mcap_path、qa_status 等必要字段。

3 调用检查器

通过 registry 调用 mcap_magic。

4 写入历史

插入 qa_checks，保留每次执行证据。

5 更新门禁

失败时设置 qa_status=failed。

检查器结果模型

type QACheckResult struct {
    CheckName string
    Passed bool
    Score float64
    Details string
    Metadata map[string]any
}

检查项注册

mcap_magic -> Go 内置检查器
sidecar_schema -> 未来检查器
python:<script_name> -> 未来脚本检查器

mcap_magic 检查细节

  • 期望 magic：89 4d 43 41 50 30 0d 0a。
  • 先通过 S3 stat 获取对象大小。
  • 对象大小小于 16 字节时直接返回 passed=false。
  • 读取 Range: bytes=0-7 作为 head magic。
  • 读取 Range: bytes=(size-8)-(size-1) 作为 tail magic。
  • head 或 tail 任一不匹配即失败。
  • 不初始化 MCAP reader，不加载索引，不解压 chunk。

后续门禁改造

行为 规则 实现点
MCAP 预览 URL qa_status=failed 时拒绝签发 EpisodeHandler.GetEpisodePresignedURL
MCAP 下载 URL qa_status=failed 时拒绝签发 kind=mcap 的 presign 请求
JSON sidecar 下载 不受 MCAP 完整性失败影响 kind=sidecar 保持现有行为
普通云同步 现有 approved 条件自然阻止 failed SyncHandler.TriggerEpisodeSync
重新同步 qa_status=failed 时阻止 SyncHandler.TriggerEpisodeResync

前端实现

前端仅在 episode 详情页增加固定按钮，不引入检查项下拉。通用化保留在 API 和后端 registry。

入口

synapse/src/views/admin/episodes/EpisodeDetail.vue 的“数据预览”卡片。

按钮

新增 MCAP 质检 按钮，点击后调用通用质检接口。

状态

展示 loading、通过提示、失败原因，并在失败后刷新详情。

API 封装

episodesApi.runQACheck(id, { check_name: 'mcap_magic' })

交互规则

  • 点击 MCAP 质检 后按钮进入提交中状态。
  • 返回 passed=true 时显示“MCAP 头尾校验通过”。
  • 返回 passed=false 时显示 details，并刷新 episode 详情。
  • 当前 episode 的 qaStatus === 'failed' 时，前端可禁用 MCAP 预览和下载按钮。
  • 后端仍是最终门禁，前端禁用只是体验优化。

测试计划

Keystone

  • 有效 MCAP：head/tail 均匹配，写入 qa_checks.passed=true。
  • head 不匹配：写入失败记录并更新 qa_status=failed。
  • tail 不匹配：写入失败记录并更新 qa_status=failed。
  • 文件小于 16 字节：失败并记录文件大小。
  • S3 不可用：返回基础设施错误，不误更新 qa_status。
  • presign?kind=mcap 在 failed 时拒绝。
  • presign?kind=sidecar 在 failed 时仍允许。
  • 同步和重新同步在 failed 时拒绝。

Synapse

  • 详情页显示 MCAP 质检 按钮。
  • 点击按钮时展示提交中状态。
  • 通过后显示成功提示。
  • 失败后显示失败原因并刷新 QA 状态。
  • qaStatus=failed 时 MCAP 预览和下载按钮禁用或显示后端错误。

后续 Python 脚本扩展

当前实现只需要 registry 和统一结果模型，不执行 Python 脚本。后续接入脚本时，脚本输出应复用同一 JSON 结构，Go 侧只负责执行、解析、落库和状态更新。

已确认决策

  • 不新增字段，复用 qa_status、quality_flag 和 qa_checks。
  • 所有检查项失败都把 qa_status 置为 failed。
  • 检查通过不自动批准，也不自动恢复失败状态。
  • 已经批准的 episode 质检失败后允许降级为 failed。
  • 当前前端只做固定按钮 MCAP 质检。
  • MCAP 失败阻止 MCAP 预览、下载和云同步，sidecar 下载保持允许。
