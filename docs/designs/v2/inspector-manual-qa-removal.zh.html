核心判断

这不是隐藏页面，也不是保留兼容接口。V2 是新的数据库基线，产品模型中不再存在检查员和人工复核终态；所有代码路径必须服从同一个四状态 QA 契约。

01

背景与问题

当前未提交代码已经把 Keystone 的迁移入口切到 migrations/v2 和 schema_migrations_v2，同时 Synapse 删除了 inspector 页面、路由和部分人工 QA 状态展示。这说明目标方向是新 V2 基线，而不是在旧模型上做小修。

问题在于后端运行时代码还保留旧契约：仍注册 /inspectors 路由，episode detail 仍 join inspections 与 inspectors，同步、数据运维和统计仍接受 needs_inspection、inspector_approved、rejected。这会让干净 V2 数据库和应用代码直接不匹配。

必须一次性对齐：数据库基线、后端状态机、OpenAPI、Synapse UI、测试断言。
02

目标与非目标

目标

  • 彻底移除 inspector / inspection 人工复核领域模型。
  • V2 schema 只描述当前可运行模型，不自动迁移旧库。
  • QA 状态统一为四个值：待质检、质检中、已通过、失败。
  • 云同步只允许 approved episode。
  • 前端不再展示人工 QA 状态，也不保留 inspector 功能死代码。

非目标

  • 不做旧生产数据库的自动数据迁移。
  • 不保留 /inspectors 兼容接口或 410 过渡接口。
  • 不把旧历史设计文档全部重写。
  • 不引入新的人工审核状态，如 needs_review。
  • 不把自动 QA 和人工检查字段混合复用。
03

方案总览

系统从“自动 QA + 人工复核”收敛为“自动 QA 单一路径”。qa_checks 继续记录检查事实，episodes.qa_status 只保存自动 QA 的聚合状态。人工检查员、检查决策和检查时间从数据模型与 API 契约中删除。

04

关键契约

状态映射

旧状态不会在 V2 运行模型中出现；如果离线迁移旧数据，映射规则应保持简单、可解释。

旧状态 / 旧语义 V2 状态 理由
inspector_approved approved 人工通过不再是独立终态；能同步的数据统一视为通过。
needs_inspection failed 没有人工队列后，不确定或异常必须落到自动 QA 失败。
rejected failed 拒绝数据在新模型中等价于未通过自动 QA。

V2 schema

episodes.qa_status ENUM(
  'pending_qa',
  'qa_running',
  'approved',
  'failed'
) DEFAULT 'pending_qa'

同步门禁

if episode.qa_status != 'approved' {
  reject sync request
}
05

后端修改面

删除 inspector 模块 移除 internal/api/handlers/inspector.go，从 server.go 删除 handler 初始化、字段和路由注册。
删除 inspection 响应字段 EpisodeResponse 不再暴露 inspector_id、inspection_decision、inspected_at，查询移除对应 join。
收紧 QA 状态 QA、data-ops、statistics、sync worker、sync API 统一只接受四状态；同步只认 approved。

组织删除逻辑必须同步移除对 inspectors 表的依赖，否则 V2 干净数据库删除组织时会查询不存在的表。Swagger 注释和生成文件必须跟随代码再生成，避免运行契约和文档契约分裂。

06

前端修改面

Synapse 采取彻底删除策略，不保留隐藏功能文件。删除 inspector API wrapper、页面组件、路由和侧边栏入口；确认没有残留 import。QA 状态筛选、badge、统计样式、同步按钮判断都只识别四状态模型。

pending_qa qa_running approved failed
07

权衡与风险

风险 影响 缓解措施
旧库不会自动升级 老环境直接切 V2 可能保留旧表和旧 enum，造成环境差异。 明确 V2 是新基线；旧数据迁移另开脚本，不在启动 migration 中隐式处理。
外部客户端仍调用 /inspectors 接口消失后旧客户端会失败。 OpenAPI 删除路径，发布说明中标记 breaking change；本设计不提供 410 过渡。
旧状态残留在代码路径 可能出现无法写入 enum、统计错误或 UI 状态裸显。 用搜索和测试覆盖 needs_inspection、inspector_approved、rejected 的运行代码残留。
Swagger 未同步 前端和外部集成继续按旧契约开发。 后端代码完成后运行 swag init -g internal/server/server.go -o docs。
08

实施与验证

  1. 清理 Keystone inspector handler、server 注册、组织删除依赖、episode detail inspection join 和响应字段。
  2. 将 QA、data-ops、statistics、sync API、sync worker 中的状态集合收敛到四状态，并把同步资格改为只允许 approved。
  3. 确认 migrations/v2 不创建 inspectors、inspections，并保留旧 migrations/ 目录作为历史，不参与运行。
  4. 同步清理 Synapse inspector 文件、路由、菜单、状态筛选、badge 和统计样式。
  5. 删除或改写测试：旧 inspector 测试删除，QA 失败路径期望 failed，sync 测试只允许 approved。
  6. 重新生成 Swagger/OpenAPI，验证 /inspectors 路径、人工检查字段和旧 QA 状态说明都已消失。
  7. 执行 Keystone 测试、Synapse build，并记录任何受本地 sandbox 限制无法完成的验证。

最终状态：一个 V2 基线，一个 QA 状态机，一个同步门禁。旧人工复核链路被切断，不再悬挂在运行代码里。
